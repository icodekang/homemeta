# HomeMeta 软件架构设计文档

## 1. 项目概述

### 1.1 项目背景

| 项目 | 规格 |
|------|------|
| 硬件平台 | RK3588 (ARMv8.2, 6核 Cortex-A76/A55, 6TOPS NPU) |
| 视频输入 | 6路 1080P RTSP 流 |
| AI 模型 | YOLO v8 (目标检测) + YOLO-Pose (行为识别) |
| 处理帧率 | 1 fps 降采样 |
| 应用场景 | 家庭安防监控、智能分析 |

### 1.2 设计目标

- **高性能**: 充分利用 RK3588 NPU 加速 AI 推理
- **低延迟**: 6路视频实时分析与告警
- **低功耗**: 针对嵌入式设备优化
- **可扩展**: 模块化设计，支持功能扩展
- **稳定性**: 7×24 小时稳定运行

## 3. 各模块职责

### 3.1 视频采集模块 (Video Capture)

| 职责 | 说明 |
|------|------|
| RTSP 连接管理 | 维护 6 路 RTSP 连接，支持断线重连 |
| 视频解码 | 利用 RK3588 VPU 硬件解码 (H.264/H.265) |
| 帧提取 | 1fps 降采样控制 |
| 错误处理 | 网络异常、格式错误的容错处理 |

**技术要点**:
- 使用 MediaMTX 或自定义 RTSP Client
- VPU 硬件解码降低 CPU 占用
- 双缓冲机制避免帧丢失

### 3.2 AI 推理引擎 (Inference Engine)

| 职责 | 说明 |
|------|------|
| 模型加载 | YOLO v8 + YOLO-Pose 模型管理 |
| NPU 加速 | 利用 RK3588 NPU 进行推理加速 |
| 批处理 | 多路视频批量推理提高效率 |
| 结果后处理 | NMS、置信度过滤、坐标转换 |

**技术要点**:
- RKNN (RK3588 NPU SDK) 加速推理
- INT8 量化模型减少内存和延迟
- 批处理提升 NPU 利用率

### 3.3 行为识别模块 (Behavior Recognition)

| 职责 | 说明 |
|------|------|
| 姿态估计 | 人体 17 关键点检测 |
| 行为分析 | 跌倒、徘徊、闯入等行为判断 |
| 时序分析 | 基于多帧的行为序列分析 |
| 告警触发 | 异常行为实时告警 |

### 3.4 任务调度模块 (Task Scheduler)

| 职责 |说明 |
|------|------|
| 帧率控制 | 1fps 精确控制 |
| 负载均衡 | 6 路视频负载分配 |
| 优先级队列 | 告警帧优先处理 |
| 资源管理 | CPU/NPU/内存资源分配 |

### 3.5 存储模块 (Storage)

| 职责 | 说明 |
|------|------|
| 事件存储 | 告警事件持久化 (SQLite) |
| 视频片段 | 告警前后录像存储 |
| 模型缓存 | AI 模型文件管理 |
| 日志存储 | 运行日志和统计 |

### 3.6 通信模块 (Communication)

| 职责 | 说明 |
|------|------|
| REST API | 设备配置、状态查询 |
| WebSocket | 实时告警推送 |
| 消息队列 | 模块间异步通信 |

## 6. 关键设计决策

### 6.1 为什么选择 1fps 降采样?

| 考虑因素 | 决策 |
|----------|------|
| 家庭场景特点 | 无需实时 30fps，1fps 足够检测异常 |
| NPU 负载 | 6 路 1080P @ 1fps 约需 15-20 TOPS (NPU 6TOPS 可处理) |
| 存储成本 | 视频存储空间减少 30 倍 |
| 告警准确性 | 降低误报率，提高检测质量 |

### 6.2 NPU 加速策略

**核心策略**: 利用 RK3588 NPU 进行 AI 推理加速

1. **模型量化**: FP16 → INT8 量化，推理速度提升 2-3 倍
2. **算子优化**: 使用 RKNN 优化算子 (Conv, Pooling, Softmax)
3. **内存复用**: NPU 与 DDR 直接数据传输，减少 CPU 拷贝
4. **批处理**: 多路视频批量推理，提高 NPU 利用率

### 6.3 并行架构设计

```
                    ┌─────────────┐
                    │  Main       │
                    │  Process    │
                    └──────┬──────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
        ▼                  ▼                  ▼
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│   Camera 1    │  │   Camera 2    │  │   Camera N    │
│   Handler     │  │   Handler     │  │   Handler     │
│   (Thread)    │  │   (Thread)    │  │   (Thread)    │
└───────────────┘  └───────────────┘  └───────────────┘
        │                  │                  │
        └──────────────────┼──────────────────┘
                           │
                           ▼
                ┌─────────────────────┐
                │   AI Inference      │
                │   Pool (Process)     │
                │   - Worker 1         │
                │   - Worker 2         │
                │   - Worker N         │
                └─────────────────────┘
```

**设计理由**:
- 每路视频独立线程，避免单路故障影响全局
- AI 推理使用进程池，避免 GIL 限制
- 6 路视频共用推理资源，提高 NPU 利用率

### 6.4 内存管理策略

| 策略 | 实现 |
|------|------|
| 内存池 | 预分配帧缓冲区，避免动态分配 |
| 零拷贝 | NPU 直接读取 DDR，减少数据拷贝 |
| 显存优化 | INT8 模型降低显存占用 |
| 缓存管理 | 模型缓存常驻，帧缓冲循环使用 |

### 6.5 容错与恢复

| 故障场景 | 恢复策略 |
|----------|----------|
| RTSP 断连 | 指数退避重连 (1s, 2s, 4s, 8s...) |
| NPU 异常 | 自动降级到 CPU 推理 (RKNPU fallback) |
| 内存泄漏 | 定期内存回收，进程健康检查 |
| 程序崩溃 | Watchdog 监控，自动重启 |

## 9. 总结

本架构针对 RK3588 硬件平台特性进行了深度优化:

1. **NPU 优先**: 充分利用 6TOPS NPU 加速 AI 推理
2. **并行处理**: 6 路视频独立采集，共享 AI 推理资源
3. **1fps 平衡**: 在性能和准确性之间取得最佳平衡
4. **模块化设计**: 清晰的层次结构，便于维护和扩展
5. **容错机制**: 完善的故障恢复，保证稳定运行

本设计可支撑 6 路 1080P 视频的实时 AI 分析，并预留了扩展接口支持更多路数或更复杂的 AI 场景。
